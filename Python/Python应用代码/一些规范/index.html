    一些规范

    一些规范 Commit提交规范

    提交commit需要遵循一定的规范，这样赘述问题的问题可以很好定位问题

int:初始化项目
feat:增加新的功能
fix:修改功能，修复bug
fmt:修改格式之类的，无关功能
doc:修改文档和修改注释，无关功能
test:测试代码，无关功能
file:移动文件，移除文件，无关功能

    如果有多个修改，比如即fix了，又有doc，按照排序在前的标注

    提交Commit的建议做法:

    原子化提交,即每一次提交的都是单一的功能, 不要一下子提交太多功能的修改
多Commit,少Push,每天必须要把当天的代码Push完
切换到发布tag的时候不要Commit

    版本号管理

    版本号统一使用 v20250720.02类似格式,使用小写v和数字组成

    软件部署

    软件测试前需自测通过
测试前需通知到当站人员
需备好编辑器软件,压缩软件,FTP连接软件
智能化程式统一部署在PyApp目录,不要安装在其他目录,优先D盘,不要安装在CD软盘之类,没有D盘就E盘,尽量不要C盘
GUI程式使用集成包方式部署,内嵌入Python解释器
命令行软件一定要加入开机自启,优先使用系统级开机自启,不行再使用用户级开机自启
部署新软件,老版本软件备份,安装日期备注
临时修改的软件必须回传到FTP目录,并回传到git

    编程规范

    缩进统一使用4个空格,不要使用默认的tab,但可以设定1 tab = 4 空格
软件必须支持心跳/自动更新使用utils
约定大于配置,库的导入有官方规范就按照官方的,没有按照团队的
入口文件要足够简单,直接就是导入模块,执行模块,入口函数名必须是 main
使用with 打开文件,不要手动关闭文件,且必须指定文件编码,with获取值之后要及时退出,不要一直占用文件
循环内不要写和循环本身无关的东西,能放在循环外的必须放在循环外
所有代码必须经过类型检查,不接受报红代码,无法通过类型检查的代码不允许提交,除非第三方库本身问题除外
函数参数必须增加类型注解,用于类型检查
函数必须增加注释,解释作用及参数意义,注意,不要写无意义及有歧义的内容
使用f-string方式来拼接字符串(除非你知道你不会遇到XP系统,否则使用format格式化字符串)
不要相信用户的输入,必须做一定的防呆,比如去除首尾的空行,换行等
自己创建的路径不要使用中文,不要自己给自己找麻烦
启动函数应该在 if name == “main”: 内
源代码应该带文件头,即表明版本的和编码的备注
函数名应该按照[蛇形小写形式]比如:get_file_name(),除了类和常量,不要使用大写,文件和目录也不要使用大写
尽量不要使用全局变量,除非十分必要
软件必须增加日志上传功能
