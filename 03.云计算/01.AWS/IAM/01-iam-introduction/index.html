    1. IAM 介绍#

    1.1 什么是 IAM#

    AWS Identity and Access Management（IAM）是一项 Web 服务，用于安全地控制对 AWS 资源的访问。通过 IAM 可以管理身份验证（谁可以登录）与授权（具有什么权限）。

    核心职责：

    • 管理身份（用户、角色、用户组）
    • 管理权限（策略）
    • 与其他 AWS 服务集成，提供日志与审计

    IAM 与 AWS IAM Identity Center、AWS STS 属于 AWS 账户内置功能，本身不收费，仅在通过 IAM 用户或 STS 临时凭证调用其他 AWS 服务时产生费用。

    1.2 为什么使用 IAM#

    特性 说明
    共享账户访问 无需共享密码或访问密钥即可授权他人使用账户资源
    精细权限 可对不同用户、不同资源授予不同权限
    EC2 应用安全访问 通过角色为 EC2 上运行的应用提供临时凭证
    MFA 多重身份验证 可绑定密码或访问密钥的二次验证
    联合身份 允许企业目录或第三方 IdP 用户访问 AWS
    身份日志 与 CloudTrail 集成记录请求者信息
    PCI DSS 合规 已通过支付卡行业数据安全标准认证

    1.3 何时使用 IAM#

    每次访问 AWS 账户都会使用 IAM。常见用户角色：

    • 服务用户：由管理员分发凭证和权限，使用 AWS 服务完成工作
    • 服务管理员：管理 AWS 资源，可能具备完全 IAM 访问权限
    • IAM 管理员：管理 IAM 身份，编写访问策略

    典型使用场景：

    • 以 IAM 用户、根用户身份登录，或担任 IAM 角色
    • 创建与附加策略（基于身份、基于资源、信任策略）
    • 使用 ABAC（基于属性）或 RBAC（基于角色）授权模型

    1.4 最终一致性与服务可用性#

    IAM 通过在 Amazon 全球数据中心多服务器间复制实现高可用。IAM 变更（创建/更新用户、组、角色、策略）需要时间传播，不建议将 IAM 变更放入应用程序的关键路径，应在初始化或安装程序中执行。

    2. IAM 管理方式#

    2.1 AWS 管理控制台#

    基于 Web 的图形界面，首次登录即可访问多个服务控制台。登录方式：

    • IAM 用户 / 根用户：通过 https://signin.aws.amazon.com 登录
    • IAM Identity Center 用户：通过组织专属的访问门户登录
    • OIDC / SAML 联合主体：通过自定义企业访问门户登录

    建议根用户仅在必要场景下使用，日常登录优先使用 IAM Identity Center 或 IAM 角色的临时凭证。

    2.2 AWS 命令行工具#

    两套主要 CLI：

    • AWS CLI（Command Line Interface）
    • AWS Tools for Windows PowerShell

    AWS CloudShell 允许从浏览器直接运行 CLI/SDK 命令，权限继承自控制台登录凭证。

    2.3 凭证优先级#

    CLI 或 SDK 的凭证可以从多个来源读取，优先级从高到低：

    1. 命令行参数
    2. 环境变量
    3. 本地 AWS 配置文件

    推荐凭证方案：

    • 本地开发：IAM Identity Center + IAM Roles Anywhere
    • AWS 环境运行：IAM 角色或 IAM Identity Center 凭证
    • 生产自动化：使用 aws configure sso 配置可自动刷新的 IAM Identity Center 凭证

    2.4 AWS SDK#

    官方 SDK 覆盖多种语言（Java、Python、Ruby、.NET、iOS、Android 等），提供：

    • 请求加密签名
    • 错误管理与自动重试
    • IAM 管理 API 封装

    2.5 IAM Query API#

    直接通过 HTTPS 请求调用 IAM，需要自行对请求进行数字签名（SigV4）。

    3. IAM 工作原理#

    3.1 身份验证流程#

    1. 用户或应用使用凭证登录 AWS
    2. IAM 匹配凭证与受信任主体（IAM 用户、STS 联合主体、IAM 角色、应用）
    3. 主体对资源发起请求
    4. IAM 根据策略评估决定允许或拒绝

    3.2 请求的组成部分#

    请求上下文包括：

    • 操作：主体希望执行的操作（控制台操作或 API/CLI 调用）
    • 资源：操作针对的 AWS 对象
    • 主体：发起请求的用户或应用（附带权限策略）
    • 环境数据：IP 地址、用户代理、SSL 状态、时间戳
    • 资源数据：资源相关元信息（如 DynamoDB 表名、EC2 标签）

    3.3 主体身份验证方式#

    主体类型 身份验证方式
    根用户 账户邮箱 + 密码
    联合主体 由外部 IdP 认证并传递凭证到 AWS
    IAM Identity Center 用户 通过 AWS 访问门户提交用户名密码
    IAM 用户 账户 ID/别名 + 用户名 + 密码；API/CLI 使用访问密钥或 STS 临时凭证

    建议对所有用户启用 MFA。

    3.4 授权基础#

    IAM 将大部分权限策略以 JSON 文档形式存储。评估逻辑：

    • 默认拒绝所有请求（根用户访问自身账户资源通常被允许）
    • 任何权限策略中的显式允许覆盖默认拒绝
    • SCP（服务控制策略）、RCP（资源控制策略）、权限边界、会话策略必须都允许才能通过
    • 任何策略中的显式拒绝覆盖所有允许

    条件元素（Condition）可以根据请求属性（如时间、IP、标签）动态控制访问。

    4. IAM 身份与凭证#

    4.1 IAM 管理的身份#

    • IAM 用户
    • IAM 用户组
    • IAM 角色
    • 权限策略
    • 身份提供者对象

    强烈建议不要用根用户执行日常任务。可以添加 IAM Identity Center 用户、联合 IdP 用户或最低权限 IAM 用户代替。

    4.2 核心术语#

    术语 含义
    IAM 资源 IAM 服务存储的对象（用户、组、角色、策略、IdP）
    IAM 实体 AWS 用于身份验证的资源（IAM 用户、IAM 角色）
    IAM 身份 策略中授权的 IAM 资源（用户、组、角色）
    主体 可向 AWS 发起请求的根用户、IAM 用户、IAM 角色
    人类用户 人员、管理员、开发者、操作员等
    工作负载 应用、进程、工具等资源和代码
    联合主体 由外部 IdP（Active Directory、Okta、Entra 等）管理的用户

    4.3 IAM 用户 vs IAM Identity Center 用户#

    对比项 IAM 用户 IAM Identity Center 用户
    凭证类型 长期凭证（密码、访问密钥） 短期临时凭证
    登录方式 账户 ID + 用户名 + 密码 组织访问门户
    是否自动担任角色 否 登录即自动担任 IAM 角色
    推荐场景 无法使用角色的工作负载、第三方客户端、服务特定凭证 集中式人员身份管理

    4.4 联合身份适用场景#

    • 公司目录与 SAML 2.0 兼容：配置 SSO 访问 AWS 管理控制台
    • 公司目录不兼容 SAML：创建身份代理程序应用
    • Microsoft Active Directory：通过 IAM Identity Center 连接 AD 或 AWS Directory Service
    • 外部 IdP（Okta、Entra）：通过 IAM Identity Center 建立信任
    • Web/移动应用：与 Login with Amazon、Facebook、Google 或 OIDC 兼容的 IdP 联合（推荐使用 Amazon Cognito）

    4.5 编程访问凭证选择#

    用户类型 推荐方案
    IAM Identity Center 用户 短期凭证签名 CLI/API 请求
    IAM 用户（推荐） 临时凭证（访问密钥 ID + 秘密密钥 + 会话令牌）
    IAM 用户（不推荐） 长期访问密钥
    联合主体 调用 AWS STS API 创建临时安全凭证会话

    5. 权限和策略#

    5.1 策略类型概览#

    策略是定义权限的 JSON 对象，附加到 IAM 身份或 AWS 资源。

    基于身份的策略（附加到 IAM 用户、组、角色）：

    • AWS 托管策略：由 AWS 创建和管理，新手推荐
    • 客户管理型策略：由用户在账户中创建和管理，更精细
    • 内联策略：嵌入到单个身份中，通常不推荐

    基于资源的策略（附加到资源）：

    • 资源策略：如 S3 存储桶策略
    • IAM 角色信任策略：定义哪些主体可以担任该角色
    • 跨账户访问：资源策略中的 Principal 可以指定其他账户

    5.2 策略示例#

    允许对指定 DynamoDB 表执行所有操作：

    {
  "Version": "2012-10-17",
  "Statement": {
    "Effect": "Allow",
    "Action": "dynamodb:*",
    "Resource": "arn:aws:dynamodb:us-east-2:123456789012:table/Books"
  }
}

    未显式允许的操作或资源均被默认拒绝。多个策略的权限叠加生效。

    5.3 角色的信任策略#

    IAM 角色同时是身份和资源，因此需要：

    • 信任策略：规定哪些主体可以担任
    • 基于身份的策略（内联或托管）：规定角色可以对资源执行的操作

    6. ABAC 授权模型#

    6.1 ABAC 简介#

    基于属性的访问控制（Attribute-Based Access Control）通过标签（Tag）定义权限。将标签附加到 IAM 实体和 AWS 资源，当主体标签与资源标签匹配时允许操作。

    6.2 ABAC vs RBAC#

    模型 思路 优缺点
    RBAC 按工作职能编写策略，列举资源 新资源必须更新策略，维护成本高
    ABAC 按属性匹配授权，动态适应变化 无需为每个资源或人员调整策略

    6.3 ABAC 优势#

    • 随创新扩展：新资源只要带上正确标签即自动获得对应权限
    • 策略数量更少：不需要为每个职能撰写独立策略
    • 动态响应变化：新增项目或人员调动只需改标签
    • 精细权限：基于标签匹配实现最小权限
    • 联合属性支持：可配置 SAML/OIDC 提供商传递会话标签给 IAM

    6.4 示例#

    创建三个角色，使用同一 access-project 标签键，值分别为 Heart、Star、Lightning。一条 ABAC 策略即可覆盖三个项目，新增项目时仅需新建带新标签的角色即可。
