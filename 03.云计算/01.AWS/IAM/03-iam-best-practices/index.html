    1. IAM 安全防御最佳实操#

    以下实践是通用建议，需结合具体环境判断是否适用。

    1.1 人类用户使用联合身份 + 临时凭证#

    • 人类用户（员工、开发者、管理员、合作者）应通过身份提供商使用临时凭证
    • 集中管理推荐 AWS IAM Identity Center
    • 身份可来自 IAM Identity Center 自有目录，或来自外部 IdP（Okta、Entra、AD）
    • 临时凭证通过代入 IAM 角色获得

    1.2 工作负载使用 IAM 角色临时凭证#

    AWS 内部计算资源（EC2、Lambda 等）由 AWS 自动为其分配临时凭证。

    AWS 外部工作负载（本地服务器、其他云、CI/CD 平台）获取临时凭证的方式：

    场景 方案
    基于 X.509 PKI IAM Roles Anywhere
    SAML IdP AssumeRoleWithSAML
    OIDC JWT AssumeRoleWithWebIdentity
    物联网设备 AWS IoT Core MTLS
    容器 Amazon ECS Anywhere
    Kubernetes Amazon EKS Hybrid Nodes
    SSM 管理 AWS Systems Manager Hybrid Activations

    1.3 强制多重身份验证（MFA）#

    • 优先使用抗网络钓鱼的 MFA：通行密钥、FIDO 安全密钥
    • 所有账户类型的根用户必须启用 MFA，首次登录 35 天内注册
    • IAM Identity Center 配合身份存储、AWS Managed Microsoft AD、AD Connector 可用 IAM Identity Center MFA

    1.4 长期凭证定期更新#

    • 尽可能用临时凭证替代访问密钥等长期凭证
    • 离职、密钥泄露等场景必须轮换
    • 使用 IAM 上次使用信息辅助判断是否可安全删除

    必须使用长期凭证的场景：

    • 无法使用 IAM 角色的工作负载（如 WordPress 插件）
    • 第三方 AWS 客户端
    • CodeCommit 访问（推荐改用 IAM Identity Center + git-remote-codecommit）
    • Amazon Keyspaces 兼容性测试

    1.5 保护根用户凭证#

    详见第 2 章。

    1.6 最低权限原则#

    先从较宽权限入手，在熟悉工作负载后逐步收紧。借助 IAM Access Analyzer 基于 CloudTrail 生成精细策略。

    1.7 起步使用 AWS 托管策略#

    • AWS 托管策略适用于多数常见场景，用于快速起步
    • 长期使用需转向客户管理型策略以达到最低权限
    • 工作职能的 AWS 托管策略覆盖 Administrator、Billing、Developer、NetworkAdmin、DatabaseAdmin、SystemAdmin、Support 等

    1.8 定期清理未使用对象#

    通过上次访问信息识别并删除：未使用的用户、角色、权限、策略、凭证。

    1.9 策略条件加固访问#

    在策略 Condition 元素中添加限制，例如：

    • 强制 TLS 请求
    • 仅允许通过 CloudFormation 访问
    • 特定 IP 范围、时间段、MFA 状态

    1.10 使用 IAM Access Analyzer 审查#

    • 验证对资源的公共和跨账户访问，生成调查发现
    • 持续监控支持的资源类型
    • 提供 100+ 策略检查，辅助策略语法与安全性验证

    1.11 多账户权限护栏#

    • 使用 AWS Organizations 服务控制策略（SCP）：限制账户内主体的操作
    • 使用资源控制策略（RCP）：限制组织中 AWS 资源的访问
    • SCP 与 RCP 不授予权限，仍需结合身份/资源策略

    1.12 权限边界委派#

    • 权限边界使用托管策略设定 IAM 角色/用户的最大权限
    • 本身不授予权限
    • 用于将权限管理安全地委派给下级团队

    2. 根用户最佳实践#

    2.1 基本原则#

    • 创建管理用户处理日常任务，根用户只做必须的根任务
    • 组织内：成员账户建议移除根用户凭证并停用 MFA，实现集中管理
    • 独立账户：使用 IAM 角色代替长期 IAM 用户
    • 多账户：使用 IAM Identity Center 员工用户

    2.2 凭证保护#

    • 不共享根用户密码、MFA、访问密钥、CloudFront 密钥对或签名证书
    • 不要使用依赖同一 AWS 账户的工具存储该账户的根密码
    • 考虑存储位置的多人审批与日志监控

    2.3 强密码要求#

    AWS 对根密码要求：

    • 长度 8–128 字符
    • 至少包含以下三类字符的组合：大写字母、小写字母、数字、符号（! @ # $ % ^ & * ( ) < > [ ] { } | _ + - =）
    • 不得与账户名称或邮箱相同

    2.4 MFA 配置#

    • 根用户可注册最多 8 台支持类型的任意组合
    • 支持类型：FIDO 硬件安全密钥、TOTP 硬件令牌、虚拟 MFA 应用
    • 启用多个 MFA 提升弹性

    2.5 不要为根用户创建访问密钥#

    • 根用户可访问所有服务和账单
    • 编程访问建议使用 aws login 命令获取临时凭证
    • 确需根凭证的任务建议通过控制台完成

    2.6 多人审批登录#

    将 MFA 与密码拆分由不同管理员持有，任一根登录需两类管理员同时参与。

    2.7 使用组电子邮件地址#

    用业务管理的邮箱并转发给一组人，避免个人休假/离职导致联系延误。该邮箱不得用于其他用途。

    2.8 账户恢复#

    • 保持根邮箱收件箱可访问，以便重置密码
    • 多个 MFA 作为备份
    • 保持注册电话号码最新，用于 MFA 恢复
    • 邮箱和电话号码不得由同一人掌控

    2.9 AWS Organizations 的根用户保护#

    两种策略（二选一）：

    1. 集中根访问：移除成员账户根用户凭证（密码、访问密钥、签名证书、MFA）
    2. 启用 MFA：为每个成员账户根用户启用 MFA

    管理账户根用户必须注册 MFA（35 天宽限期）。

    2.10 使用 SCP 限制根用户操作#

    在启用根用户凭证的成员账户中，可使用 SCP 拒绝除必要根用户操作以外的所有根用户操作。

    2.11 监控与告警#

    • CloudTrail 日志：记录根用户登录事件和 sts:AssumeRoot 特权会话操作
    • CloudWatch：检测根用户登录并发送通知
    • EventBridge + SNS：针对特定根用户操作发出提醒
    • GuardDuty：威胁检测扩展到根用户活动

    提醒应包括根用户使用的邮箱地址，并配套应急响应流程。

    2.12 MFA 合规评估#

    工具 功能
    AWS Config 托管规则监控根 MFA 和访问密钥
    Security Hub CSPM 安全状态视图，评估是否符合 MFA、无访问密钥等标准
    Trusted Advisor 安全检查是否启用根 MFA

    若按最佳实践移除了成员账户的根用户凭证，则 MFA 规则会显示为不适用或不合规（这是预期行为）。

    3. 典型商用案例#

    3.1 初始设置：Example Corp 场景#

    创始人 Nikki 与 Mateo 建立 AWS 账户后：

    1. 启用 IAM Identity Center 管理管理员账户
    2. 不再使用根用户登录，仅用于需根凭证的任务
    3. 根用户启用 MFA，并设置凭证保护流程
    4. 员工通过 Active Directory 集成 IAM Identity Center 统一管理身份
    5. 使用工作职能的 AWS 托管策略创建权限集：Administrator、Billing、Developer、NetworkAdmin、DatabaseAdmin、SystemAdmin、Support
    6. 权限集通过 IAM Identity Center 映射到 AD 组

    3.2 IAM 与 Amazon EC2 集成#

    周边控制：

    • 在 AllUsers 组附加策略，拒绝来自企业网络外部 IP 的请求

    分组权限：

    用户组 权限策略 说明
    SysAdmins AmazonEC2FullAccess 完全 EC2 操作
    Developers 自定义允许 DescribeInstances、RunInstances、StopInstances、StartInstances、TerminateInstances 仅能使用实例
    Support 自定义允许 EC2 Describe* 仅查看不操作

    注意：IAM 策略无法控制登录 EC2 实例操作系统。操作系统访问需依赖 SSH 密钥、Windows 密码、安全组。

    职能变更：当开发人员 Paulo 转为管理人员加入 Support 组，将失去实例启动停止终止权限，仅能列出实例。

    3.3 IAM 与 Amazon S3 集成#

    桶结构（使用键前缀划分）：

    /amzn-s3-demo-bucket
    /home
        /zhang
        /major
    /share
        /developers
        /managers

    策略分配：

    • Zhang 用户：读、写、列出 /amzn-s3-demo-bucket/home/zhang/ 下对象
    • Mary 用户：读、写、列出 /amzn-s3-demo-bucket/home/major/ 下对象
    • Developers 组：读、写、列出 /amzn-s3-demo-bucket/share/developers/ 下对象
    • Managers 组：读、写、列出 /amzn-s3-demo-bucket/share/managers/ 下对象

    关键点：S3 不会因创建对象自动授予后续操作权限，需在策略中显式赋予。

    职能变更：Zhang 从 Developers 转到 Managers 组后自动失去 share/developers/ 访问权限。

    第三方合作：为合作伙伴 Widget Company 创建独立用户组 WidgetCo 与专用桶 amzn-s3-demo-bucket1。可通过附加 Deny 策略限制只能执行写入操作。
