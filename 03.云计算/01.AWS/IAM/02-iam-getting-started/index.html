    1. IAM 入门#

    IAM 是 AWS 账户内置功能，使用免费。访问 AWS 资源前需要完成账户与身份初始化。

    1.1 账户初始化要点#

    注册 AWS 账户会生成一个根用户（email + 密码）。核心原则：

    • 根用户只用于必须使用根凭证的任务
    • 日常操作通过 IAM Identity Center 用户、IAM 用户或联合身份执行
    • 根用户必须启用 MFA

    1.2 激活账单控制台访问权限#

    默认情况下 IAM 用户与角色即使拥有 Billing 相关策略也不能访问账单和成本管理控制台，需要根用户激活：

    1. 使用根用户登录控制台
    2. 导航栏选择账户名称 → 账户
    3. 滚动至 IAM 用户和角色的账单信息访问权限 → 编辑
    4. 勾选 Activate IAM Access → 更新

    激活只是前提，仍需为用户或角色附加账单策略（如 Billing 托管策略）才能实际访问。

    2. 查看与管理账户 ID#

    2.1 查看账户 ID#

    控制台：IAM 控制面板的 AWS 账户部分或右上角用户名下方。

    CLI：

    aws sts get-caller-identity

    API：GetCallerIdentity。

    2.2 账户别名#

    账户 ID 是 12 位数字，创建别名后的登录 URL 更容易记忆：

    默认 URL：

    https://Your_Account_ID.signin.aws.amazon.com/console/

    使用别名后：

    https://Your_Account_Alias.signin.aws.amazon.com/console/

    别名规则：

    • 每个账户仅允许一个别名，新建将覆盖旧别名
    • 只能包含数字、小写字母、连字符
    • 在分区内全 AWS 产品范围内唯一
    • 别名不是密码，将公开显示在登录 URL 中
    • 原始账户 ID URL 在创建别名后依然有效

    支持的分区：

    • aws：AWS 标准区域
    • aws-cn：中国区域
    • aws-us-gov：AWS GovCloud (US) 区域

    2.3 创建/删除账户别名操作#

    所需权限：iam:ListAccountAliases、iam:CreateAccountAlias、iam:DeleteAccountAlias。

    CLI 操作：

    # 创建别名
aws iam create-account-alias --account-alias my-alias

# 删除别名
aws iam delete-account-alias --account-alias my-alias

# 查看当前别名
aws iam list-account-aliases

    删除别名后基于账户 ID 的 URL 仍可用，原别名 URL 失效。

    3. 规划访问权限#

    3.1 身份来源选择#

    优先使用临时凭证。推荐方案：

    • 启用 IAM Identity Center + AWS Organizations，在组织目录中直接添加用户
    • 联合现有外部 IdP（Okta、Entra、AD）到 IAM Identity Center 或 IAM

    3.2 访问管理配置#

    • 使用 IAM Identity Center：自动在每个成员账户创建 IdP、IAM 角色和权限策略
    • 直接联合 IdP 到 IAM：需手动创建角色 + 信任策略 + 权限策略

    4. IAM 用户的使用场景#

    IAM 用户拥有长期凭证，不建议作为默认方案。仅在以下场景使用：

    • 紧急访问（IdP 不可用时的后备）
    • 无法使用 IAM 角色的工作负载
    • AWS CodeCommit 访问
    • Amazon Keyspaces 访问
    • 第三方 AWS 客户端
    • 没有 IAM Identity Center 或其他 IdP 的账户

    4.1 创建紧急访问 IAM 用户#

    控制台流程：

    1. IAM 控制台 → 用户 → 创建用户
    2. 用户名如 EmergencyAccess
    3. 勾选向 AWS 管理控制台 提供用户访问权限 + 我想创建一个 IAM 用户
    4. 选择自动生成密码，不勾选下次登录时创建新密码
    5. 将用户添加到新组 EmergencyAccessGroup，附加 AdministratorAccess 托管策略
    6. 下载包含凭证的 .csv 文件

    用户名规则：

    • 长度不超过 64 字符
    • 允许：字母、数字、+ = , . @ _ -
    • 账户内唯一，不区分大小写
    • 在策略/ARN 中区分大小写，控制台显示不区分大小写

    4.2 CLI 创建紧急访问用户#

    # 创建用户
aws iam create-user --user-name EmergencyAccess

# 生成 CLI 输入模板
aws iam create-login-profile --generate-cli-skeleton > create-login-profile.json

# 编辑 JSON 后执行
aws iam create-login-profile --cli-input-json file://create-login-profile.json

# 创建组并附加管理员策略
aws iam create-group --group-name EmergencyAccessGroup
aws iam attach-group-policy \
    --policy-arn arn:aws:iam::aws:policy/AdministratorAccess \
    --group-name EmergencyAccessGroup

# 将用户加入组
aws iam add-user-to-group \
    --user-name EmergencyAccess \
    --group-name EmergencyAccessGroup

# 查看组成员
aws iam get-group --group-name EmergencyAccessGroup

    create-login-profile.json 示例：

    {
  "UserName": "EmergencyAccess",
  "Password": "Ex@3dRA0djs",
  "PasswordResetRequired": false
}

    密码不符合策略会报错 PasswordPolicyViolation。

    4.3 为工作负载创建 IAM 用户#

    适用于无法担任 IAM 角色的程序。关键点：

    • 提供编程访问凭证（访问密钥、SSH 公钥、Git 凭证、服务特定凭证、X.509 证书）
    • 建议使用 PowerUserAccess 托管策略授予除 IAM 和 Organizations 之外的全部权限
    • 遵循最低权限原则，考虑更限制性的自定义策略
    • 访问密钥仅在创建时显示一次，必须妥善保存

    凭证类型：

    类型 用途
    访问密钥 API、CLI 长期凭证
    SSH 公有密钥 CodeCommit SSH 通信
    HTTPS Git 凭证 CodeCommit HTTPS 通信
    Amazon Keyspaces 服务凭证 Keyspaces 访问
    X.509 签名证书 SOAP 协议请求（ACM 不支持区域）

    CLI 示例：

    # 创建用户与组
aws iam create-user --user-name Automation
aws iam create-group --group-name AutomationGroup
aws iam attach-group-policy \
    --policy-arn arn:aws:iam::aws:policy/PowerUserAccess \
    --group-name AutomationGroup
aws iam add-user-to-group \
    --user-name Automation \
    --group-name AutomationGroup

# 创建访问密钥
aws iam create-access-key --user-name Automation

# 上传 SSH 公钥
aws iam upload-ssh-public-key \
    --user-name Automation \
    --ssh-public-key-body file://sshkey.pub

# 上传 X.509 签名证书
aws iam upload-signing-certificate \
    --user-name Automation \
    --certificate-body file://certificate.pem

    如果担心访问密钥风险，可以启用 MFA 并用 aws sts get-session-token 获取短期会话凭证代替长期密钥。

    5. 多重身份验证（MFA）#

    • 强烈建议为根用户、IAM 用户、IAM Identity Center 用户、Builder ID、联合用户启用
    • 所有账户类型的根用户都要求配置 MFA，首次登录后 35 天内必须注册
    • IAM Identity Center 默认开启 MFA
    • 可通过策略强制要求 MFA 后才能访问特定资源或操作

    6. 最低权限许可#

    6.1 实施思路#

    • 根用户、管理员、紧急访问用户拥有过高权限，不应用于日常任务
    • 在 IAM Identity Center 创建至少一个低权限用户用于日常
    • 使用 IAM Identity Center 权限集或 IAM 角色来组织权限

    6.2 渐进式最低权限#

    AWS 托管策略可能权限过宽，推荐流程：

    1. 先使用 AWS 托管策略快速起步
    2. 观察一段采样期（如 90 天）的访问活动
    3. 使用 IAM Access Analyzer 基于 CloudTrail 事件生成最低权限策略
    4. 创建客户管理型策略替换 AWS 托管策略

    7. 上次访问信息#

    7.1 支持的资源#

    资源 上次访问信息内容
    IAM 用户 用户最近尝试访问每个允许服务的时间
    IAM 组 成员最近访问时间与尝试访问的总人数
    IAM 角色 使用角色最近尝试访问每个服务的时间
    策略 使用策略访问服务的实体与最后时间

    7.2 跟踪周期#

    • 服务跟踪周期：过去 400 天
    • S3 操作：从 2020-04-12 开始
    • EC2、Lambda、IAM 操作：从 2021-04-07 开始
    • 其他所有服务：从 2023-05-23 开始

    7.3 控制台操作#

    路径：IAM 控制台 → 用户/组/角色/策略 → 对应资源 → 上次访问时间选项卡。

    上次访问时间列的可能结果：

    • <数量> 天前：在跟踪周期内的访问天数
    • 在跟踪周期间未访问：期间内未使用

    7.4 CLI / API 操作#

    # 生成报告（需指定 IAM 资源 ARN），返回 job-id
aws iam generate-service-last-accessed-details --arn <iam-arn>

# 查询服务级结果
aws iam get-service-last-accessed-details --job-id <job-id>

# 查询使用策略/组的实体详情
aws iam get-service-last-accessed-details-with-entities --job-id <job-id> --service-namespace <ns>

# 列出给定身份访问指定服务所使用的策略
aws iam list-policies-granting-service-access \
    --arn <identity-arn> --service-namespaces <ns1> <ns2>

    对应 API：GenerateServiceLastAccessedDetails、GetServiceLastAccessedDetails、GetServiceLastAccessedDetailsWithEntities、ListPoliciesGrantingServiceAccess。

    该操作仅返回基于身份的策略，不涵盖资源策略、ACL、Organizations 策略、权限边界、会话策略。

    8. 基于访问活动生成策略#

    8.1 两种生成结果#

    • 操作级别策略：对支持的服务（如 EC2）列出具体的 Action
    • 服务级别策略：基于上次访问信息生成包含服务名称的模板，需手动补完 Action

    8.2 控制台生成步骤#

    1. IAM 控制台 → 用户 → 详情页 → 权限选项卡
    2. 基于 CloudTrail 事件生成策略 → 生成策略
    3. 选择时间段（如过去 7 天）、选择 CloudTrail 跟踪所在的区域与跟踪
    4. 选择创建和使用新的服务角色
    5. 生成完成后审查和自定义策略
    6. 根据需要调整 Action、Resource、账户 ID
    7. 使用 IAM Access Analyzer 策略验证解决错误和警告
    8. 保存策略并附加到目标角色，移除原 AWS 托管策略

    精简后的策略示例：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "FullAccessToSomeServices",
      "Effect": "Allow",
      "Action": ["ec2:*", "s3:*"],
      "Resource": "*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "iam:CreateServiceLinkedRole",
        "iam:DeleteServiceLinkedRole",
        "iam:ListRoles",
        "organizations:DescribeOrganization"
      ],
      "Resource": "*"
    }
  ]
}

    8.3 CLI 操作#

    # 启动策略生成
aws accessanalyzer start-policy-generation --policy-generation-details '{...}'

# 查看生成的策略
aws accessanalyzer get-generated-policy --job-id <id>

# 取消策略生成
aws accessanalyzer cancel-policy-generation --job-id <id>

# 查看策略生成请求列表
aws accessanalyzer list-policy-generations

    9. IAM 控制台搜索#

    功能：

    • 查找匹配关键字的 IAM 实体名称（用户、组、角色、IdP、策略）
    • 查找匹配的任务链接
    • 不返回 Access Analyzer 信息
    • 访问密钥搜索需输入完整访问密钥 ID
