    1. 策略与权限概述#

    IAM 通过策略（JSON 文档）附加到身份或资源来管理访问。主体发起请求时，AWS 评估适用策略决定允许或拒绝。

    1.1 评估基本规则#

    账户内评估顺序：

    1. 所有请求默认隐式拒绝（根用户访问自身账户资源通常允许）
    2. 身份或资源策略中的显式允许覆盖默认
    3. 权限边界、SCP、RCP、会话策略可用隐式拒绝覆盖允许
    4. 任何策略的显式拒绝覆盖所有允许

    1.2 策略类型#

    类型 附加位置 作用 是否授予权限
    基于身份的策略 用户、组、角色 控制身份可执行的操作 是
    基于资源的策略 资源（S3 桶、SQS、角色信任策略等） 控制主体对资源的访问 是
    权限边界 IAM 用户、IAM 角色 限制身份策略能授予的最大权限 否（只限制）
    Organizations SCP 组织、OU、账户 限制账户主体的最大权限 否
    Organizations RCP 组织、OU、账户 限制账户资源的最大权限 否
    ACL 资源 控制跨账户主体访问（S3、WAF、VPC） 是
    会话策略 角色/联合会话运行时 限制本次会话权限 否（只限制）

    1.3 跨账户访问特殊规则#

    • 基于资源的策略支持跨账户主体
    • 跨账户访问同时需要：目标账户资源策略允许 + 源账户身份策略允许
    • 同账户内仅一边允许即可

    1.4 基于身份策略进一步分类#

    • AWS 托管策略：AWS 创建与维护，可直接附加
    • 客户托管策略：用户自建，可版本化与回滚
    • 内联策略：嵌入单个身份，策略随身份生命周期

    2. JSON 策略结构#

    2.1 策略文档#

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "OptionalStatementId",
      "Effect": "Allow | Deny",
      "Principal": { ... },
      "Action": "...",
      "Resource": "...",
      "Condition": { ... }
    }
  ]
}

    多个策略与语句之间是逻辑 OR（合集效果，但 Deny 优先）。

    2.2 元素说明#

    元素 说明
    Version 推荐 2012-10-17（与策略元素 Version 区分）
    Statement 语句数组
    Sid 可选语句标识
    Effect Allow 或 Deny
    Principal / NotPrincipal 仅基于资源的策略（或信任策略）需要
    Action / NotAction 允许/拒绝的操作列表
    Resource / NotResource 操作适用的资源 ARN
    Condition 可选条件，如 MFA、IP、时间、标签

    2.3 多语句示例#

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "FirstStatement",
      "Effect": "Allow",
      "Action": ["iam:ChangePassword"],
      "Resource": "*"
    },
    {
      "Sid": "SecondStatement",
      "Effect": "Allow",
      "Action": "s3:ListAllMyBuckets",
      "Resource": "*"
    },
    {
      "Sid": "ThirdStatement",
      "Effect": "Allow",
      "Action": ["s3:List*", "s3:Get*"],
      "Resource": [
        "arn:aws:s3:::amzn-s3-demo-bucket-confidential-data",
        "arn:aws:s3:::amzn-s3-demo-bucket-confidential-data/*"
      ],
      "Condition": {
        "Bool": { "aws:MultiFactorAuthPresent": "true" }
      }
    }
  ]
}

    2.4 基于资源策略示例（S3 桶策略）#

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "1",
      "Effect": "Allow",
      "Principal": {
        "AWS": ["arn:aws:iam::111122223333:root"]
      },
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::amzn-s3-demo-bucket",
        "arn:aws:s3:::amzn-s3-demo-bucket/*"
      ]
    }
  ]
}

    3. 根用户与策略#

    • 不能为根用户附加基于身份的策略或权限边界
    • 基于资源策略和 ACL 可将根用户列为主体
    • 根用户属于账户，受 Organizations SCP 和 RCP 约束

    4. 最低权限实践#

    1. 访问级别分组：策略操作分为 List、Read、Write、Permissions management、Tagging
    2. 策略验证：使用 IAM Access Analyzer 策略验证（100+ 检查项）
    3. 基于访问活动生成策略：根据 CloudTrail 日志生成模板
    4. 上次访问信息：定位从未使用的权限
    5. CloudTrail 事件历史：精确定位实际使用的 Action 和 Resource

    5. 托管策略 vs 内联策略#

    对比 AWS 托管策略 客户托管策略 内联策略
    管理者 AWS 用户 用户
    可变 否 是 是
    附加数量 多个实体 多个实体 单个实体
    自动更新 是 否 否
    版本控制 是 最多 5 版 否

    5.1 AWS 托管策略类型#

    • 完全访问：AmazonDynamoDBFullAccess、IAMFullAccess
    • 高级用户：AWSCodeCommitPowerUser、AWSKeyManagementServicePowerUser
    • 部分访问：AmazonEC2ReadOnlyAccess、AmazonMobileAnalyticsWriteOnlyAccess
    • 工作职能：AdministratorAccess、PowerUserAccess、Billing、DatabaseAdministrator、NetworkAdministrator 等

    ARN 格式：arn:aws:iam::aws:policy/<policy-name>。

    5.2 何时选择#

    推荐场景 策略类型
    快速起步、通用场景 AWS 托管策略
    最小权限自定义、多实体复用 客户托管策略
    一次性使用、严格一对一关系、身份删除即移除 内联策略

    5.3 托管策略优势#

    • 可重用：同一策略附加多个主体
    • 集中变更：更新一次自动生效所有附加主体
    • 版本回滚：客户托管策略保留 5 个版本
    • 委派管理：可允许受限管理员只能附加特定策略
    • 更大字符数上限：比内联策略限制宽

    5.4 将内联策略转换为托管策略#

    1. IAM 控制台 → 用户/组/角色 → 权限选项卡
    2. 复制内联策略的 JSON
    3. 新建客户托管策略并粘贴
    4. 附加到目标身份
    5. 删除原内联策略

    6. 策略大小与配额#

    • 客户管理型策略：最多 6144 字符
    • 内联策略每个身份：用户 2048、组 5120、角色 10240 字符
    • 每个角色最多 20 个内联策略（默认）
    • 客户托管策略每账户最多 1500 个
    • 托管策略附加到一个身份上限：10

    7. 策略与 AWS CLI/API 使用#

    CLI：

    # 创建策略
aws iam create-policy --policy-name <name> --policy-document file://p.json

# 附加到身份
aws iam attach-user-policy --user-name <u> --policy-arn <arn>
aws iam attach-group-policy --group-name <g> --policy-arn <arn>
aws iam attach-role-policy --role-name <r> --policy-arn <arn>

# 更新策略（创建新版本）
aws iam create-policy-version --policy-arn <arn> --policy-document file://p.json --set-as-default
aws iam list-policy-versions --policy-arn <arn>
aws iam set-default-policy-version --policy-arn <arn> --version-id v2
aws iam delete-policy-version --policy-arn <arn> --version-id v1

# 内联策略
aws iam put-user-policy --user-name <u> --policy-name <n> --policy-document file://p.json
aws iam put-group-policy --group-name <g> ...
aws iam put-role-policy --role-name <r> ...

aws iam list-user-policies --user-name <u>
aws iam get-user-policy --user-name <u> --policy-name <n>
aws iam delete-user-policy --user-name <u> --policy-name <n>

    7.1 策略 ARN 格式#

    • AWS 托管：arn:aws:iam::aws:policy/<name>
    • 客户托管：arn:aws:iam::<account-id>:policy/<name>
    • 工作职能：arn:aws:iam::aws:policy/job-function/<name>

    8. 策略元素：条件键（Condition）#

    8.1 常用全局条件键#

    条件键 说明
    aws:CurrentTime 请求时间
    aws:EpochTime 请求时间（Unix 时间戳）
    aws:SecureTransport 是否使用 HTTPS
    aws:SourceIp 请求源 IP
    aws:SourceVpc / aws:SourceVpce 源 VPC 或 VPC 端点
    aws:MultiFactorAuthPresent 是否使用 MFA
    aws:MultiFactorAuthAge MFA 完成至今秒数
    aws:PrincipalArn / aws:PrincipalType 主体 ARN 或类型
    aws:PrincipalAccount / aws:PrincipalOrgID / aws:PrincipalOrgPaths 主体账户或组织
    aws:PrincipalTag/<key> 主体标签
    aws:RequestTag/<key> 请求携带的标签
    aws:ResourceTag/<key> 资源标签（用于 ABAC）
    aws:TagKeys 请求中的标签键集合
    aws:CalledVia / aws:CalledViaFirst / aws:CalledViaLast 调用链（服务代表用户调用的服务）
    aws:ViaAWSService 请求是否经 AWS 服务
    aws:TokenIssueTime 临时凭证签发时间（撤销会话）
    aws:UserAgent 请求 User-Agent

    8.2 常用运算符#

    类型 运算符
    字符串 StringEquals、StringNotEquals、StringEqualsIgnoreCase、StringLike、StringNotLike
    数字 NumericEquals、NumericLessThan 等
    日期 DateGreaterThan、DateLessThan 等
    布尔 Bool
    IP 地址 IpAddress、NotIpAddress
    ARN ArnEquals、ArnLike
    空值 Null
    多值集合 ForAllValues:<op>、ForAnyValue:<op>
    存在性条件 <op>IfExists（若条件键存在则匹配，否则跳过）

    8.3 常用模式#

    仅 HTTPS：

    "Condition": {
  "Bool": { "aws:SecureTransport": "true" }
}

    限制 IP：

    "Condition": {
  "IpAddress": { "aws:SourceIp": ["203.0.113.0/24", "2001:DB8:1234:5678::/64"] }
}

    要求 MFA：

    "Condition": {
  "Bool": { "aws:MultiFactorAuthPresent": "true" },
  "NumericLessThan": { "aws:MultiFactorAuthAge": "3600" }
}

    ABAC 标签匹配：

    "Condition": {
  "StringEquals": {
    "aws:ResourceTag/Project": "${aws:PrincipalTag/Project}"
  }
}

    限制服务内调用（FAS）：

    "Condition": {
  "ForAllValues:StringEquals": {
    "aws:CalledVia": ["cloudformation.amazonaws.com"]
  }
}

    9. 策略变量#

    策略中可使用 ${...} 引用请求上下文变量：

    • ${aws:username}
    • ${aws:userid}
    • ${aws:PrincipalTag/<key>}
    • ${aws:SourceIp}
    • ${saml:sub}
    • ${token.actions.githubusercontent.com:sub}（OIDC 声明）

    示例：允许用户访问以其用户名为前缀的 S3 对象：

    {
  "Effect": "Allow",
  "Action": ["s3:GetObject", "s3:PutObject"],
  "Resource": "arn:aws:s3:::amzn-s3-demo-bucket/home/${aws:username}/*"
}

    10. 跨账户资源访问#

    10.1 模式#

    • 代理模式（推荐）：角色信任其他账户，基于资源策略最小化，通过角色完成跨账户
    • 直接资源策略：S3 桶策略、KMS 密钥策略等直接允许其他账户主体

    10.2 跨账户必备条件#

    • 资源策略允许跨账户主体
    • 主体所在账户身份策略允许相应操作
    • 关联的 SCP、RCP、权限边界未阻止

    10.3 转发访问会话（FAS）#

    AWS 服务代表用户调用其他服务时使用主体凭证 + 服务主体信息（不是服务角色）。可用 aws:CalledVia 等键识别与限制 FAS 请求。

    10.4 PassRole#

    将 IAM 角色传递给 AWS 服务（如 EC2 启动实例关联角色）需要 iam:PassRole 权限。典型策略：

    {
  "Effect": "Allow",
  "Action": "iam:PassRole",
  "Resource": "arn:aws:iam::<account-id>:role/EC2AppRole",
  "Condition": {
    "StringEquals": { "iam:PassedToService": "ec2.amazonaws.com" }
  }
}

    11. AWS 数据边界#

    组合 Organizations SCP/RCP、VPC 端点策略、IAM 策略，实现身份边界、资源边界、网络边界，防止未授权的跨身份/资源/网络访问。

    • 身份边界：限制组织内身份能访问的资源
    • 资源边界：限制组织内资源能被哪些身份访问
    • 网络边界：限制资源只能通过受信任网络访问

    相关参考：AWS SRA 安全参考架构。
