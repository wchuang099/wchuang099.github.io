    1. 常见问题速查#

    1.1 无法登录账户#

    先区分用户类型：

    • 根用户：使用账户邮箱和创建时的密码
    • IAM 用户：账户 ID/别名 + 用户名 + 密码
    • IAM Identity Center 用户：组织专属访问门户

    参考 AWS 登录用户指南的登录故障排查。

    1.2 丢失访问密钥#

    • 访问密钥 ID 公开，可在 IAM 控制台查看
    • 秘密密钥仅创建时可见，丢失无法恢复
    • 必须删除旧密钥并创建新的（每用户最多两个）

    1.3 策略变量不生效#

    常见原因：

    • Version 元素未设置为 2012-10-17
    • 变量大小写错误（例：${AWS:USERNAME} 错误）
    • 在不支持的位置使用变量

    1.4 最终一致性延迟#

    IAM 通过全球数据中心复制，创建或更新用户、组、角色、策略、标签后可能需要几秒到数分钟才能在所有端点生效。

    建议：

    • 不要把 IAM 变更放在应用关键路径
    • 初始化流程或管理后台执行
    • 生产部署依赖前验证变更已传播

    1.5 虚拟 MFA 分配失败错误#

    is not authorized to perform: iam:DeleteVirtualMFADevice
on resource: ... with an explicit deny

    原因：之前启动 MFA 分配流程时取消，遗留了未分配的虚拟 MFA 设备。解决：

    aws iam list-virtual-mfa-devices
aws iam delete-virtual-mfa-device --serial-number <arn>

    1.6 安全创建 IAM 用户的工作流#

    1. 控制台创建用户，选自动生成密码 + 要求首次登录修改
    2. 复制登录 URL、用户名、密码
    3. 通过公司安全渠道传递凭证，URL 与用户名单独发送
    4. 员工确认可以登录后再附加权限
    5. 强制 MFA 管理自身凭证

    2. 访问被拒绝错误解读#

    错误格式：

    User: <principal-arn> is not authorized to perform: <action> on resource: <resource>
because <context>

    根据 <context> 定位问题。分隐式拒绝（没有允许语句）和显式拒绝（有 Deny 语句）。

    2.1 常见拒绝类型#

    上下文提示 位置 处理方法
    no service control policy allows the action SCP 缺允许 在组织 SCP 添加 Allow
    with an explicit deny in a service control policy SCP Deny 修改 SCP
    with an explicit deny in a resource control policy RCP Deny 修改 RCP
    no VPC endpoint policy allows VPCe 策略缺允许 更新 VPC 端点策略
    with an explicit deny in a VPC endpoint policy VPCe Deny 修改 VPC 端点策略
    no permissions boundary allows 权限边界未允许 更新边界策略
    with an explicit deny in a permissions boundary 权限边界 Deny 修改权限边界
    no session policy allows 会话策略缺允许 传递包含 Allow 的会话策略
    with an explicit deny in a session policy 会话策略 Deny 调整传递的策略
    no resource-based policy allows 资源策略缺允许 修改资源策略
    with an explicit deny in a resource-based policy 资源策略 Deny 修改资源策略
    no role trust policy allows 信任策略缺允许 修改角色信任策略
    with an explicit deny in the role trust policy 信任策略 Deny 修改角色信任策略
    no identity-based policy allows 身份策略缺允许 附加或修改基于身份的策略
    with an explicit deny in an identity-based policy 身份策略 Deny 修改身份策略

    2.2 使用临时凭证被拒绝#

    排查清单：

    1. 凭证是否过期（查看 Expiration）
    2. 是否传递了限制性会话策略
    3. 请求签名是否正确（手动签名时容易出错）
    4. IAM 用户或角色的基础权限是否允许
    5. 如果跨账户且 VPC 端点调用，VPC 端点账户与角色账户是否一致

    2.3 条件键大小写陷阱#

    条件键名称不区分大小写。foo、Foo、FOO 都匹配同一条件。传入多条键值对时不要仅用大小写区分。

    3. 策略问题排查#

    3.1 可视化编辑器行为#

    • 策略先经 AWS 处理后存储，再读取时结构可能被调整（重排、压缩）
    • 不要用字符串对比 JSON 策略
    • 仅在 JSON 编辑器中编辑且保存，结构保持原样

    3.2 选择资源 ARN#

    • ARN 生成器：按资源类型填字段，未知值留空或选任意（通配）
    • 直接粘贴 ARN：字段内可使用 * 通配
    • 任何位置不建议使用跨字段通配（冒号之间）

    3.3 拒绝权限建议#

    • 默认拒绝，因此无需为尚未允许的操作加 Deny
    • 仅当需要覆盖其他允许时才使用 Deny
    • 过多 Deny 会增加调试难度

    3.4 多服务策略#

    • 单条语句可通过通配指定："Action": "Code*"
    • 但必须相应填写跨服务的 Action 与 Resource
    • 建议分语句组织，便于维护

    3.5 策略未授予预期权限#

    排查顺序：

    1. 检查是否同时满足基于身份 + 基于资源策略（跨账户）
    2. 检查权限边界、SCP/RCP
    3. 检查会话策略（临时凭证）
    4. 检查 Condition 是否被满足（时间、IP、MFA、标签）
    5. 使用 IAM 策略模拟器或 simulate-principal-policy 逐项验证
    6. 查看 CloudTrail 事件获取真实错误上下文

    3.6 JSON 策略常见问题#

    • 多个策略对象：一个文档只能有一个顶层 {}，其他语句放入 Statement 数组
    • 多个 Statement 元素：只能有一个 Statement 键
    • 同一 Statement 包含多个 Effect、Action、Resource：使用数组或分 Statement
    • 缺少 Version 或写成 "Version": "2008-10-17" 导致变量不解析
    • 策略大小超过限制需拆分

    3.7 策略摘要问题#

    • 缺少摘要：通常新创建未保存时；或策略语法错误
    • 包含无法识别的服务/操作：说明使用了过时或错拼的名称
    • 服务不支持策略摘要：参见 AWS 文档列表

    4. 根用户故障#

    4.1 忘记根密码#

    • 使用账户邮箱发起密码重置
    • 必须能访问邮箱收件箱

    4.2 无法访问根邮箱#

    • 如果拥有域名可恢复邮箱或设置 catch-all
    • 公司邮箱联系 IT 系统管理员
    • 其他情况通过 AWS Support 联系

    4.3 根用户被 SCP 限制#

    组织管理员可能设置 SCP 限制了根用户操作。联系管理账户管理员协调。

    4.4 成员账户根用户不存在#

    启用集中根访问后，成员账户的根凭证可能被主动移除。需要通过委托管理员发起特权任务或恢复密码。

    5. 角色故障#

    5.1 无法担任角色#

    • 信任策略是否包含要担任的主体
    • 权限策略是否允许 sts:AssumeRole
    • 外部 ID 是否匹配
    • 是否要求 MFA 但未提供
    • 角色会话时长是否超上限
    • 会话策略是否拒绝

    5.2 服务无法担任角色#

    • 服务主体 ARN 是否正确（区分大小写）
    • iam:PassRole 权限是否具备
    • 跨服务访问时 aws:SourceArn、aws:SourceAccount 条件是否匹配

    5.3 无法编辑角色#

    • 服务相关角色权限由服务管理，IAM 管理员只能查看
    • 检查当前身份是否有 iam:UpdateRole、iam:PutRolePolicy 等权限

    6. EC2 和 IAM 常见问题#

    6.1 无法启动指定角色的实例#

    • 角色必须拥有对应实例配置文件，或实例配置文件与角色关联
    • 当前身份需要 iam:PassRole 且条件 iam:PassedToService 为 ec2.amazonaws.com
    • IAM 变更后可能存在延迟（最终一致性）

    6.2 EC2 实例无法访问资源#

    • 登录实例检查元数据：curl http://169.254.169.254/latest/meta-data/iam/security-credentials/<role>
    • 确认附加的角色与 aws sts get-caller-identity 返回一致
    • 检查目标服务的资源策略和 KMS 密钥策略

    7. S3 与 IAM 问题#

    7.1 桶策略被拒#

    常见原因：

    • 桶策略拒绝所有主体（配置错误）
    • 依赖 aws:SourceIp 但请求来自 VPC 端点，应同时允许 aws:SourceVpc/aws:SourceVpce
    • KMS 加密桶时，调用方还需 KMS 密钥权限

    7.2 解锁桶策略#

    • 桶拥有者根用户可以强制编辑/删除策略
    • Organizations 场景下：通过委派管理员启动 sts:AssumeRoot 特权会话并附加 S3UnlockBucketPolicy 任务策略

    8. SAML 联合故障#

    8.1 Invalid SAML response#

    • 元数据文件含 BOM 或编码错误
    • X.509 证书密钥长度不足或含重复扩展
    • IdP 返回的签名与元数据不匹配

    8.2 signature mismatch#

    • 证书链顺序错误：应为域/颁发者证书 → 中间证书 → 根证书
    • 存在重复或无关证书

    8.3 权限类错误#

    • RoleSessionName 含非法字符或空格（应 2–64，字母数字 + . , + = @ -）
    • saml:aud 未匹配到 https://signin.aws.amazon.com/saml
    • IdP 发送的 AWS/Role 属性值不是合法的 Role ARN / Provider ARN

    8.4 查看 SAML 响应#

    浏览器开发者工具 → Network → 查找 POST 请求中的 SAMLResponse（Base64）。PowerShell 解码：

    [System.Text.Encoding]::UTF8.GetString([System.Convert]::FromBase64String("<base64>"))

    9. 通行密钥与 FIDO 问题#

    • FIDO 密钥不同步，无法通过 ResyncMFADevice 修复，需管理员禁用并重新注册
    • Chrome 弹窗不要选择“用 amazon.com 验证”，直接触碰安全密钥
    • 支持的浏览器：Chrome、Edge、Safari 等主流浏览器；参考 AWS WebAuthn 兼容配置列表

    10. 排查工具#

    • IAM 控制台策略验证：创建或编辑策略时的实时提示
    • IAM Access Analyzer：验证、自定义检查、策略生成
    • IAM 策略模拟器：线上模拟策略评估
    • CloudTrail：查看请求上下文、所使用的策略、真实拒绝源
    • 最近访问信息：定位长期未使用的权限、策略
    • 凭证报告：账户级凭证状态概览
    • AWS Config：持续监控根 MFA、IAM 配置合规
    • Security Hub CSPM：整合 IAM 相关合规控制
