    1. IAM 角色概述#

    IAM 角色是一种具有权限策略的 AWS 身份，与 IAM 用户不同的地方在于：

    • 不唯一关联到个人，任何符合条件的实体都可以担任
    • 没有长期凭证（密码、访问密钥），担任后返回临时安全凭证
    • 担任角色后原有权限暂停，退出时恢复

    使用角色的典型原因：

    • 向 AWS 账户内或跨账户的用户/应用/服务授权
    • 不把 AWS 密钥嵌入移动应用
    • 让外部身份（企业目录、OIDC/SAML）获取 AWS 访问权限
    • 授权第三方审核账户

    2. 角色术语#

    术语 含义
    角色 可被 AWS 用户、服务、联合身份代入的 IAM 身份
    AWS 服务角色 被服务担任并代表您行事的角色
    AWS 服务相关角色 与特定 AWS 服务关联、由服务拥有的预定义服务角色，IAM 管理员可查看但不能改权限
    角色链 使用一个角色代入另一个角色（会话最长 1 小时）
    委派 在两个账户之间建立信任以授权访问资源
    信任策略 角色上的基于资源的策略，规定谁可以担任
    联合身份 来自外部 IdP 的身份，通过角色访问 AWS
    权限边界 限制基于身份策略可授予的最大权限（不适用于服务相关角色）

    2.1 角色链注意事项#

    • 控制台、CLI、API 使用角色链时会话上限为 1 小时
    • AssumeRole 的 DurationSeconds 在角色链中若超过 1 小时会失败
    • 角色链中可传递会话标签（需设为可传递）

    2.2 角色可以由谁代入#

    • 同账户或跨账户的 IAM 用户
    • 同账户中的其他 IAM 角色
    • AWS 服务主体（EC2、Lambda、S3 对象复制等）
    • IAM Roles Anywhere、ECS Anywhere 等为 AWS 外部应用提供凭证的服务
    • SAML 2.0 / OIDC 外部 IdP 认证的用户

    3. 服务相关角色（SLR）特殊行为#

    • 启动服务时可能被服务自动创建
    • 权限不能手动编辑，由服务维护
    • 删除 SLR 前必须先删除相关资源
    • 计入账户 IAM 角色配额，但已达上限仍允许创建 SLR
    • 使用角色时权限可能间接传递给可调用该服务的用户（例如 RDS SLR 会间接访问 EC2、SNS 等）

    4. 混淆代理人问题#

    4.1 跨账户场景#

    第三方拥有的账户可能被其他客户提供的角色 ARN 诱导去操作您的资源。解决方案是在角色信任策略中要求 sts:ExternalId：

    {
  "Version": "2012-10-17",
  "Statement": {
    "Effect": "Allow",
    "Principal": { "AWS": "<ThirdParty-Account-ID>" },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringEquals": { "sts:ExternalId": "12345" }
    }
  }
}

    ExternalId 由第三方为每个客户生成且对其客户不透明，客户无法伪造。

    4.2 跨服务场景#

    AWS 服务主体代为访问您的资源，可能被未授权角色欺骗。使用以下全局条件键缩小范围：

    条件键 作用
    aws:SourceArn 限制特定源资源的 ARN
    aws:SourceAccount 限制特定源账户
    aws:SourceOrgID 限制特定 AWS Organizations
    aws:SourceOrgPaths 限制特定组织路径

    示例：仅允许来自指定账户的 CloudTrail 写入 S3 桶：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "CloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::amzn-s3-demo-bucket1",
      "Condition": {
        "StringEquals": { "aws:SourceAccount": "111122223333" }
      }
    },
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::amzn-s3-demo-bucket1/AWSLogs/111122223333/*",
      "Condition": {
        "StringEquals": { "aws:SourceAccount": "111122223333" }
      }
    }
  ]
}

    4.3 RCP 组织级防护#

    使用资源控制策略统一在组织层面加上跨服务防护（避免每个资源都写条件）：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "RCPEnforceConfusedDeputyProtectionForS3",
      "Effect": "Deny",
      "Principal": "*",
      "Action": ["s3:*"],
      "Resource": "*",
      "Condition": {
        "StringNotEqualsIfExists": {
          "aws:SourceOrgID": "o-ExampleOrg"
        },
        "Null": { "aws:SourceAccount": "false" },
        "Bool": { "aws:PrincipalIsAWSService": "true" }
      }
    }
  ]
}

    5. 常见角色场景#

    5.1 跨 AWS 账户访问#

    在生产账户创建 UpdateApp 角色，信任开发账户，授予对 productionapp S3 桶的读写权限。开发账户中的用户通过 sts:AssumeRole 切换到该角色使用权限。

    核心权限分工：

    • 角色所在账户：信任策略 + 权限策略
    • 使用方账户：允许 sts:AssumeRole 到目标角色 ARN 的策略

    角色 ARN 形式：arn:aws:iam::123456789012:role/UpdateApp。

    5.2 非 AWS 工作负载#

    使用 IAM Roles Anywhere，基于 X.509 证书身份验证：

    1. 使用 AWS Private CA 或自有 PKI 的 CA
    2. 将 CA 证书作为信任锚注册到 IAM Roles Anywhere
    3. 创建/复用角色并授予权限
    4. 在工作负载上用证书获取临时凭证

    5.3 第三方账户访问#

    第三方需要角色 ARN 才能担任。务必在信任策略中加入 sts:ExternalId，并设置最小权限。

    5.4 AWS 服务访问#

    创建 AWS 服务角色，服务（如 EC2、Lambda、CodeBuild）通过服务主体担任。实例需要通过实例配置文件（Instance Profile）关联。

    5.5 联合身份#

    为 SAML 或 OIDC 提供商创建对应的 IdP 和角色。联合主体登录后通过 AssumeRoleWithSAML 或 AssumeRoleWithWebIdentity 获取临时凭证。

    6. 创建 IAM 角色#

    6.1 控制台（以跨账户角色为例）#

    1. IAM 控制台 → 角色 → 创建角色
    2. 受信任实体类型：选择 AWS 账户
    3. 填写受信账户 ID；可选 ExternalId、要求 MFA
    4. 选择要附加的权限策略
    5. （可选）设置权限边界、标签
    6. 命名、审查并创建

    6.2 CLI#

    跨账户角色：

    # 1. 创建角色（指定信任策略 JSON 文件）
aws iam create-role \
    --role-name UpdateApp \
    --assume-role-policy-document file://trust.json

# 2. 附加托管权限策略
aws iam attach-role-policy \
    --role-name UpdateApp \
    --policy-arn arn:aws:iam::aws:policy/AmazonS3FullAccess

    服务角色同样使用 create-role + attach-role-policy，但信任策略的 Principal 为 {"Service":"<service>.amazonaws.com"}。

    6.3 API#

    CreateRole → AttachRolePolicy。可选 PutRolePermissionsBoundary、PutRolePolicy 设置内联策略。

    6.4 CloudFormation#

    使用 AWS::IAM::Role 资源类型，AssumeRolePolicyDocument 定义信任关系，Policies/ManagedPolicyArns 定义权限。

    6.5 信任策略示例#

    AWS 账户受信：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::123456789012:root" },
      "Action": "sts:AssumeRole"
    }
  ]
}

    AWS 服务受信：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}

    SAML 联合受信：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Federated": "arn:aws:iam::123456789012:saml-provider/ExampleCorp" },
      "Action": "sts:AssumeRoleWithSAML",
      "Condition": {
        "StringEquals": { "SAML:aud": "https://signin.aws.amazon.com/saml" }
      }
    }
  ]
}

    OIDC Web 联合（如 GitHub Actions）：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Federated": "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com" },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
        },
        "StringLike": {
          "token.actions.githubusercontent.com:sub": "repo:org/repo:*"
        }
      }
    }
  ]
}

    7. 实例配置文件（EC2）#

    EC2 实例不能直接担任角色，需通过实例配置文件：

    aws iam create-instance-profile --instance-profile-name MyAppProfile
aws iam add-role-to-instance-profile \
    --instance-profile-name MyAppProfile --role-name MyAppRole

    启动实例时指定 --iam-instance-profile Name=MyAppProfile。控制台通过角色创建时会自动创建同名实例配置文件。

    8. 创建服务相关角色#

    多数 SLR 在首次使用服务或执行特定操作时由 AWS 自动创建。也可手动创建：

    CLI：

    aws iam create-service-linked-role --aws-service-name <service>.amazonaws.com

    API：CreateServiceLinkedRole。

    9. 管理角色#

    9.1 查看角色#

    aws iam list-roles
aws iam get-role --role-name <name>
aws iam list-role-policies --role-name <name>
aws iam list-attached-role-policies --role-name <name>

    9.2 修改权限#

    # 附加托管策略
aws iam attach-role-policy --role-name <n> --policy-arn <arn>
aws iam detach-role-policy --role-name <n> --policy-arn <arn>

# 内联策略
aws iam put-role-policy --role-name <n> --policy-name <pn> \
    --policy-document file://p.json
aws iam delete-role-policy --role-name <n> --policy-name <pn>

    9.3 修改信任策略#

    aws iam update-assume-role-policy --role-name <n> --policy-document file://trust.json

    9.4 修改最大会话时长#

    CLI：

    aws iam update-role --role-name <n> --max-session-duration 7200

    默认 1 小时，可设置 1–12 小时。角色链场景下上限仍为 1 小时。

    9.5 权限边界#

    aws iam put-role-permissions-boundary --role-name <n> --permissions-boundary <arn>
aws iam delete-role-permissions-boundary --role-name <n>

    10. 删除角色#

    10.1 删除前准备#

    • 移除关联的实例配置文件
    • 分离所有托管策略
    • 删除所有内联策略
    • 删除权限边界
    • 等待依赖服务释放角色

    10.2 删除 IAM 角色#

    # 移除实例配置文件关联
aws iam remove-role-from-instance-profile \
    --instance-profile-name <ipn> --role-name <n>

# 分离与删除策略
aws iam list-attached-role-policies --role-name <n>
aws iam detach-role-policy --role-name <n> --policy-arn <arn>
aws iam list-role-policies --role-name <n>
aws iam delete-role-policy --role-name <n> --policy-name <pn>

# 删除角色
aws iam delete-role --role-name <n>

    10.3 删除服务相关角色#

    aws iam delete-service-linked-role --role-name <slr-name>
aws iam get-service-linked-role-deletion-status --deletion-task-id <id>

    服务会异步验证 SLR 是否仍被使用；若资源未删除，删除会失败。

    11. 担任角色（切换角色）#

    11.1 控制台#

    1. 导航栏 → 用户名菜单 → 切换角色
    2. 填写账户 ID/别名、角色名
    3. 可选显示名称、颜色
    4. 切换成功后右上角显示新角色，可退回原身份
    5. 最近使用过的角色会缓存以便快速切换

    注意：根用户不能切换角色；要求 ExternalId 的角色无法通过控制台切换。

    11.2 AWS CLI#

    aws sts assume-role \
    --role-arn arn:aws:iam::999999999999:role/UpdateApp \
    --role-session-name MySession \
    --duration-seconds 3600 \
    --external-id MyExtId

    返回后将凭证设置为环境变量：

    export AWS_ACCESS_KEY_ID=...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...

    也可在 ~/.aws/config 配置 source_profile 与 role_arn，命令自动完成切换。

    11.3 AWS API#

    • AssumeRole：使用 IAM 凭证（长期或短期）
    • AssumeRoleWithSAML：SAML 2.0 IdP
    • AssumeRoleWithWebIdentity：OIDC/JWT
    • AssumeRoot：委托管理员对成员账户执行特权任务

    返回的 Credentials 包含 AccessKeyId、SecretAccessKey、SessionToken、Expiration。

    12. 向用户授予切换角色的权限#

    附加到 IAM 用户或组的策略：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::999999999999:role/UpdateApp"
    }
  ]
}

    结合 aws:MultiFactorAuthPresent 可强制 MFA 才能切换。

    13. 角色会话权限扩展#

    13.1 会话策略#

    调用 AssumeRole* 时传递 Policy 或 PolicyArns 参数，用于进一步缩减角色权限（不能超过角色本身权限）。

    13.2 会话标签#

    传递 Tags 参数为当前会话附加标签；设置 TransitiveTagKeys 让标签随后续链继续生效。用于 ABAC 联合场景。
