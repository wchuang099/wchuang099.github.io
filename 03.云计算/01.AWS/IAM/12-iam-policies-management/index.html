    1. 创建策略#

    1.1 控制台三种方式#

    • 可视化编辑器：无需 JSON，按服务选择 Action 和 Resource
    • JSON 编辑器：手写或粘贴 JSON 文档
    • 导入托管策略：以已有策略为模板再自定义

    创建时 IAM 自动检测 JSON 语法错误，IAM Access Analyzer 进一步提供 100+ 项检查与建议。

    1.2 CLI 创建#

    # 客户托管策略
aws iam create-policy --policy-name MyPolicy \
    --policy-document file://p.json \
    --description "..."

# 策略验证
aws accessanalyzer validate-policy \
    --policy-document file://p.json \
    --policy-type IDENTITY_POLICY

    1.3 API#

    • CreatePolicy（客户托管）
    • PutUserPolicy / PutGroupPolicy / PutRolePolicy（内联）
    • ValidatePolicy（IAM Access Analyzer）

    2. 附加策略#

    控制台：用户/组/角色 → 权限选项卡 → 添加权限 → 附加策略。

    CLI：

    aws iam attach-user-policy --user-name <u> --policy-arn <arn>
aws iam attach-group-policy --group-name <g> --policy-arn <arn>
aws iam attach-role-policy --role-name <r> --policy-arn <arn>

aws iam list-attached-user-policies --user-name <u>
aws iam list-attached-group-policies --group-name <g>
aws iam list-attached-role-policies --role-name <r>

    API：AttachUserPolicy、AttachGroupPolicy、AttachRolePolicy、ListEntitiesForPolicy。

    3. 版本控制#

    • 客户托管策略最多保留 5 个版本
    • 新建版本时会提示是否设为默认
    • CLI 超过 5 个版本需先删除旧版本；控制台会自动提示删除
    aws iam create-policy-version --policy-arn <arn> \
    --policy-document file://p.json --set-as-default
aws iam list-policy-versions --policy-arn <arn>
aws iam get-policy-version --policy-arn <arn> --version-id v2
aws iam set-default-policy-version --policy-arn <arn> --version-id v1
aws iam delete-policy-version --policy-arn <arn> --version-id v1

    策略文档中的 "Version": "2012-10-17" 是策略语言版本，与策略版本不同。

    4. 编辑策略#

    • AWS 托管策略不能编辑
    • 客户托管策略：创建新版本并可选设为默认
    • 内联策略：直接替换文档

    CLI 示例：

    # 修改客户托管策略 = 建新版本
aws iam create-policy-version --policy-arn <arn> \
    --policy-document file://p.json --set-as-default

# 修改内联策略 = 覆盖上传
aws iam put-role-policy --role-name <r> \
    --policy-name <pn> --policy-document file://p.json

    5. 删除策略#

    5.1 客户托管策略删除#

    必须先：

    1. 从所有附加实体（用户、组、角色）分离策略
    2. 删除除默认外的所有版本
    3. 删除策略
    # 分离
aws iam list-entities-for-policy --policy-arn <arn>
aws iam detach-user-policy --user-name <u> --policy-arn <arn>
aws iam detach-group-policy --group-name <g> --policy-arn <arn>
aws iam detach-role-policy --role-name <r> --policy-arn <arn>

# 删除非默认版本
aws iam delete-policy-version --policy-arn <arn> --version-id v1

# 删除策略
aws iam delete-policy --policy-arn <arn>

    5.2 内联策略删除#

    aws iam delete-user-policy --user-name <u> --policy-name <n>
aws iam delete-group-policy --group-name <g> --policy-name <n>
aws iam delete-role-policy --role-name <r> --policy-name <n>

    6. 权限边界#

    6.1 概念#

    权限边界是一种使用托管策略约束 IAM 实体能被基于身份策略授予的最大权限。权限边界本身不授予权限，必须配合身份策略生效。

    有效权限 = 身份策略 ∩ 权限边界（身份策略和权限边界都允许才生效）。任一策略的显式拒绝有效。

    适用对象：

    • IAM 用户
    • IAM 角色（不适用于服务相关角色）

    6.2 CLI 操作#

    # 设置或更新权限边界
aws iam put-user-permissions-boundary --user-name <u> --permissions-boundary <arn>
aws iam put-role-permissions-boundary --role-name <r> --permissions-boundary <arn>

# 查看
aws iam get-user --user-name <u>
aws iam get-role --role-name <r>

# 删除
aws iam delete-user-permissions-boundary --user-name <u>
aws iam delete-role-permissions-boundary --role-name <r>

    6.3 典型用途：委派权限管理#

    允许开发团队创建 IAM 角色，但限制他们只能创建边界在某个范围内的角色：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCreateRolesWithBoundary",
      "Effect": "Allow",
      "Action": ["iam:CreateRole", "iam:AttachRolePolicy", "iam:DetachRolePolicy"],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "iam:PermissionsBoundary": "arn:aws:iam::111122223333:policy/DevTeamBoundary"
        }
      }
    },
    {
      "Sid": "DenyRemoveBoundary",
      "Effect": "Deny",
      "Action": ["iam:DeleteRolePermissionsBoundary", "iam:PutRolePermissionsBoundary"],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "iam:PermissionsBoundary": "arn:aws:iam::111122223333:policy/DevTeamBoundary"
        }
      }
    }
  ]
}

    6.4 基于资源策略与权限边界#

    • 若基于资源策略在 Principal 指定角色会话 ARN 或用户：不受权限边界限制
    • 若指定角色 ARN：需要权限边界显式允许

    7. 查看与访问策略#

    7.1 查看附加策略#

    控制台：身份详情 → 权限选项卡。可查看托管与内联策略。

    CLI：

    aws iam get-policy --policy-arn <arn>
aws iam list-policies --scope Local --only-attached
aws iam list-attached-role-policies --role-name <r>
aws iam list-role-policies --role-name <r>
aws iam get-role-policy --role-name <r> --policy-name <pn>

    7.2 控制对策略的访问#

    可通过 Condition 限制只能附加/分离特定策略：

    {
  "Effect": "Allow",
  "Action": ["iam:AttachRolePolicy", "iam:DetachRolePolicy"],
  "Resource": "*",
  "Condition": {
    "ArnEquals": {
      "iam:PolicyARN": [
        "arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess",
        "arn:aws:iam::111122223333:policy/CustomPolicy"
      ]
    }
  }
}

    8. 策略摘要与访问级别#

    控制台策略详情会显示访问级别摘要：

    访问级别 含义
    List 枚举资源
    Read 查看内容
    Tagging 添加/删除/列出标签
    Write 创建/修改/删除资源
    Permissions management 修改访问权限，如 PutBucketPolicy、PassRole

    查看策略摘要时重点关注 Write 与 Permissions management 级别。

    9. 策略验证#

    IAM Access Analyzer 在创建、保存、编辑时执行检查，分类：

    • Security warnings：可能降低安全性的语法（例如开放给所有人的 * 主体）
    • Errors：非法语法，AWS 无法理解
    • Warnings：与 AWS 推荐惯例不一致
    • Suggestions：非必需但可优化

    9.1 CLI 验证#

    aws accessanalyzer validate-policy \
    --policy-type IDENTITY_POLICY \
    --policy-document file://p.json

    策略类型：IDENTITY_POLICY、RESOURCE_POLICY、SERVICE_CONTROL_POLICY、RESOURCE_CONTROL_POLICY。

    10. 测试策略#

    10.1 IAM 策略模拟器#

    控制台地址：https://policysim.aws.amazon.com/

    功能：

    • 选择用户、组、角色
    • 选择要测试的服务、操作、资源
    • 模拟评估结果（允许或拒绝及依据）
    • 测试账户 SCP 和权限边界效果

    10.2 CLI 模拟#

    # 模拟用户策略
aws iam simulate-principal-policy \
    --policy-source-arn <user-or-role-arn> \
    --action-names s3:GetObject s3:PutObject \
    --resource-arns arn:aws:s3:::my-bucket/*

# 模拟自定义策略集
aws iam simulate-custom-policy \
    --policy-input-list file://p.json \
    --action-names s3:GetObject

    11. AWS 托管策略版本化#

    • AWS 会在新服务或新 API 发布时更新托管策略
    • 已附加的身份自动获得新权限
    • 有时某个 AWS 托管策略会被弃用并被新策略取代，需迁移到推荐策略

    查看策略的历史变更：

    aws iam list-policy-versions --policy-arn arn:aws:iam::aws:policy/ReadOnlyAccess

    12. IAM 策略缩减范围（Scope Down）#

    从宽到窄的典型流程：

    1. 先附加 AWS 托管策略保证工作可用
    2. 观察采样周期（>30 天）
    3. 使用 IAM Access Analyzer 基于 CloudTrail 生成精细策略
    4. 校验策略后替换原宽泛策略
    5. 重复迭代

    13. 常用策略示例#

    13.1 允许用户自管理密码与密钥#

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowViewAccountInfo",
      "Effect": "Allow",
      "Action": [
        "iam:GetAccountPasswordPolicy",
        "iam:ListVirtualMFADevices"
      ],
      "Resource": "*"
    },
    {
      "Sid": "AllowManageOwnPasswords",
      "Effect": "Allow",
      "Action": ["iam:ChangePassword", "iam:GetUser"],
      "Resource": "arn:aws:iam::*:user/${aws:username}"
    },
    {
      "Sid": "AllowManageOwnAccessKeys",
      "Effect": "Allow",
      "Action": [
        "iam:CreateAccessKey",
        "iam:DeleteAccessKey",
        "iam:ListAccessKeys",
        "iam:UpdateAccessKey",
        "iam:GetAccessKeyLastUsed"
      ],
      "Resource": "arn:aws:iam::*:user/${aws:username}"
    },
    {
      "Sid": "AllowManageOwnMFA",
      "Effect": "Allow",
      "Action": [
        "iam:DeactivateMFADevice",
        "iam:EnableMFADevice",
        "iam:ListMFADevices",
        "iam:ResyncMFADevice"
      ],
      "Resource": "arn:aws:iam::*:user/${aws:username}"
    },
    {
      "Sid": "DenyAllExceptListedIfNoMFA",
      "Effect": "Deny",
      "NotAction": [
        "iam:CreateVirtualMFADevice",
        "iam:EnableMFADevice",
        "iam:GetUser",
        "iam:ListMFADevices",
        "iam:ListVirtualMFADevices",
        "iam:ResyncMFADevice",
        "sts:GetSessionToken"
      ],
      "Resource": "*",
      "Condition": {
        "BoolIfExists": { "aws:MultiFactorAuthPresent": "false" }
      }
    }
  ]
}

    13.2 限制 S3 访问到特定前缀#

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::my-bucket",
      "Condition": {
        "StringLike": {
          "s3:prefix": ["home/${aws:username}/*", "home/${aws:username}"]
        }
      }
    },
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::my-bucket/home/${aws:username}/*"
    }
  ]
}

    13.3 PassRole 给 EC2#

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "iam:PassRole",
      "Resource": "arn:aws:iam::<account-id>:role/EC2AppRole",
      "Condition": {
        "StringEquals": { "iam:PassedToService": "ec2.amazonaws.com" }
      }
    },
    {
      "Effect": "Allow",
      "Action": ["ec2:RunInstances", "ec2:DescribeInstances"],
      "Resource": "*"
    }
  ]
}

    13.4 基于 IP 的访问限制#

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "NotIpAddress": {
          "aws:SourceIp": ["203.0.113.0/24", "2001:db8:1234:5678::/64"]
        },
        "Bool": {
          "aws:ViaAWSService": "false"
        }
      }
    }
  ]
}

    13.5 要求强制 TLS#

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {
        "Bool": { "aws:SecureTransport": "false" }
      }
    }
  ]
}

    14. 已弃用策略#

    AWS 会将部分过时的托管策略标记为已弃用：

    • 无法新附加
    • 已附加实体会收到迁移通知
    • AWS 通常提供替代策略

    查看：aws iam list-policies --scope AWS --policy-usage-filter PolicyName。

    15. 资源控制策略（RCP）#

    RCP 与 SCP 类似，用于从组织层面对资源施加限制：

    • 限制 S3 桶、KMS 密钥、SQS、Secrets Manager 等资源的最大权限
    • 组织内所有账户生效
    • 通过 aws:SourceOrgID 等条件键与 FAS/跨服务访问集成
    • 支持的服务列表见 Organizations 文档

    示例：组织内 S3 资源只能被组织内的主体或已授权服务访问：

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnforceOrgIdentities",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": "*",
      "Condition": {
        "StringNotEqualsIfExists": {
          "aws:PrincipalOrgID": "o-ExampleOrg"
        },
        "BoolIfExists": {
          "aws:PrincipalIsAWSService": "false"
        }
      }
    }
  ]
}
