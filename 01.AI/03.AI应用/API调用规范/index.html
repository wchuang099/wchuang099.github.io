    大模型 API 调用规范#

    概述#

    调用大模型本质上就是发 HTTP 请求，但各厂商的接口地址（Endpoint）、认证方式（Auth）、计费规则都各不相同。目前行业存在几套主要规范，其中 OpenAI 格式已成为事实标准，大多数工具和平台都优先兼容它。

    一、OpenAI Compatible（事实标准）#

    最初由 OpenAI 定义的 Chat Completions API 格式，现在被行业广泛采用。可以理解为 AI 领域的"USB-C 接口"——大家都用同一套协议，工具和模型之间可以自由组合。

    核心三要素#

    参数 说明 示例
    Base URL API 服务地址 https://api.openai.com/v1
    API Key 认证密钥 sk-xxx...
    Model 模型名称 gpt-4o、claude-3.5-sonnet

    切换模型或平台时，通常只需要改这三个值，代码逻辑不用动。

    兼容情况#

    类别 代表 兼容程度
    原生支持 OpenAI 完全兼容（定义者）
    厂商兼容 DeepSeek、Mistral、Qwen、Moonshot 完全兼容
    部分兼容 Anthropic 有自己的格式，但多数聚合平台会转换
    聚合平台 OpenRouter、LiteLLM、OneAPI 统一转为 OpenAI 格式

    支持该标准的工具#

    类型 代表工具
    IDE Cursor、Kiro
    IDE 插件 Cline、Roo Code（VS Code 系）、Continue（VS Code + JetBrains）
    CLI 工具 Claude Code CLI、Aider
    Agent Goose
    应用框架 LangChain、LlamaIndex

    基本调用示例#

    from openai import OpenAI

client = OpenAI(
    base_url="https://api.openai.com/v1",  # 换平台改这里
    api_key="sk-xxx"                         # 换密钥改这里
)

response = client.chat.completions.create(
    model="gpt-4o",                          # 换模型改这里
    messages=[
        {"role": "user", "content": "你好"}
    ]
)

print(response.choices[0].message.content)

    切换到 DeepSeek 只需改三行：

    client = OpenAI(
    base_url="https://api.deepseek.com/v1",
    api_key="sk-deepseek-xxx"
)
# model="deepseek-chat"

    二、各厂商原生 API 速查#

    不是所有厂商都兼容 OpenAI 格式，以下是主流厂商原生 API 的关键差异。

    国际厂商#

    OpenAI（GPT 系列）#
    项目 内容
    Endpoint https://api.openai.com/v1/chat/completions
    认证方式 Header Authorization: Bearer sk-...
    注意事项 国内无法直连，需代理；新账号无海外手机号验证易被封禁
    Google（Gemini 系列）#
    项目 内容
    Endpoint https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent
    认证方式 URL 参数 ?key=AIza... 或 Header x-goog-api-key
    注意事项 自成一派的 REST 格式，与 OpenAI 不兼容，需维护独立的调用代码

    Gemini 的优势在于原生多模态和超长上下文（2M Token），但 API 格式是最大的接入障碍。

    Anthropic（Claude 系列）#
    项目 内容
    Endpoint https://api.anthropic.com/v1/messages
    认证方式 Header x-api-key（注意不是 Authorization）
    官方 SDK Python: pip install anthropic，TypeScript: npm install @anthropic-ai/sdk
    注意事项 支付门槛高，仅支持少数国家信用卡

    调用示例：

    import anthropic

client = anthropic.Anthropic(api_key="sk-ant-xxx")

message = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=1024,
    system="你是一个有帮助的助手",        # system 是独立参数，不在 messages 里
    messages=[
        {"role": "user", "content": "你好"}
    ]
)

print(message.content[0].text)              # 响应是 content blocks，不是 choices

    与 OpenAI 格式的主要区别：

    对比项 OpenAI Anthropic
    System prompt 放在 messages 数组中 独立的顶层参数
    响应结构 choices[0].message.content content[0].text
    max_tokens 可选 必填
    认证 Header Authorization: Bearer x-api-key

    国内厂商#

    阿里通义千问（Qwen）#
    项目 内容
    Endpoint https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions
    认证方式 Header Authorization: Bearer sk-...（OpenAI 兼容）
    特点 提供 OpenAI 兼容模式，可直接用 OpenAI SDK 调用；中文场景性价比高
    百度文心一言（ERNIE）#
    项目 内容
    Endpoint 需先通过 OAuth2.0 获取 Access Token，再拼接到 URL 中
    认证方式 OAuth2.0（两步：先用 API Key + Secret Key 换 Token，再用 Token 调用）
    特点 接入流程较繁琐；知识图谱增强，国内百科类问题回答准确
    智谱 AI（GLM）#
    项目 内容
    Endpoint https://open.bigmodel.cn/api/paas/v4/chat/completions
    认证方式 JWT 认证（需在本地用 API Key 生成 JWT Token）
    特点 认证方式对新手不友好，需要额外的 Token 生成步骤

    三、云厂商专有 API#

    AWS Bedrock（详解）#

    Bedrock 提供多种调用方式，不只是 Boto3：

    调用方式一览：

    方式 说明
    AWS SDK（Boto3 等） 传统方式，IAM 签名认证，支持 Python/Java/JS/.NET 等
    OpenAI 兼容端点（原生） Bedrock 官方提供 OpenAI 兼容 API（基于 Mantle 推理引擎），可直接用 OpenAI SDK 调用
    bedrock-access-gateway AWS 官方开源项目，部署一个网关将 Bedrock 转为标准 OpenAI API 格式
    IDE 插件直连 Cline 官方支持 Bedrock 配置（含企业管理员模式）；Cursor 可通过 gateway 桥接

    OpenAI 兼容调用示例：

    from openai import OpenAI

# 直接用 OpenAI SDK 调用 Bedrock（需配置 AWS 认证）
client = OpenAI(
    base_url="https://bedrock-runtime.us-east-1.amazonaws.com/v1",
    api_key="<aws-credentials>"  # 实际使用 AWS 签名认证
)

response = client.chat.completions.create(
    model="anthropic.claude-3-5-sonnet-20241022-v2:0",
    messages=[{"role": "user", "content": "你好"}]
)

    推理配置文件（Inference Profile）：

    Bedrock 的一个重要特性，用于控制请求如何路由和计费：

    类型 说明
    跨区域推理配置文件 自动将请求路由到多个 AWS 区域，提升吞吐量和可用性。支持地理级别（US/EU/APAC）和全球级别路由
    应用推理配置文件 为特定应用创建独立的配置文件，用于追踪该应用的成本和 token 用量

    使用时只需将 model 参数替换为 Inference Profile ID 即可，代码逻辑不变。这对企业多团队、多应用共用 Bedrock 时的成本管控非常有用。

    Azure OpenAI#

    项目 内容
    调用方式 OpenAI 格式 + Azure 认证体系（Azure AD），需要指定 deployment name
    特点 与 OpenAI API 高度兼容，但 base_url 和认证方式不同

    Google Vertex AI#

    项目 内容
    调用方式 Google Cloud SDK 调用，使用 Google Cloud 认证
    特点 与 Gemini API 是两套入口，Vertex AI 面向企业，Gemini API 面向开发者

    选择云厂商 API 通常是因为企业已在用对应的云生态，而非 API 格式本身的优势。

    四、工具交互协议#

    除了模型调用 API，还有一些协议定义了模型如何与外部工具交互：

    协议 说明
    Function Call OpenAI 提出，让模型在对话中调用外部函数/API。模型判断需要调用什么函数、传什么参数，由应用层执行后把结果返回给模型
    MCP（Model Context Protocol） Anthropic 提出的开放标准，统一 AI 模型连接外部工具和数据源的方式。可以理解为 Function Call 的标准化和扩展版，支持更丰富的工具发现和调用机制

    五、为什么需要聚合网关#

    当你的应用需要同时调用多个厂商的模型时（比如用 Gemini 处理长文本、GPT 做推理、Claude 写文案、Qwen 做中文润色），直接对接意味着：

    • 维护多套 SDK 和认证逻辑
    • 管理多份账单
    • 处理不同的错误格式和重试策略

    聚合网关（如 LiteLLM、OpenRouter、OneAPI）的价值就是把这些差异抹平，统一成 OpenAI 格式，让你只写一套代码调用所有模型。

    详细的聚合平台分类和选型，参考：大模型聚合平台

    六、总结#

    规范 定位 适用场景
    OpenAI Compatible 模型调用的事实标准 绝大多数场景首选
    Anthropic Messages API Claude 原生格式 直接对接 Anthropic 时
    Google Gemini API Google 自有格式 需要 Gemini 原生能力时
    国内厂商各自格式 百度 OAuth、智谱 JWT 等 直接对接国内厂商时
    云厂商专有 API 深度集成云生态 企业已在用对应云平台
    Function Call / MCP 模型与工具的交互协议 Agent、工具调用场景

    实用建议：

    • 优先学 OpenAI Compatible 格式，覆盖 90% 的场景
    • 需要多模型混用时，用聚合网关而不是逐个对接
    • 遇到特定平台再查对应文档即可
